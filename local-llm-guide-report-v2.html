📌 核心概念

🎯

Token

模型的工作单位。不是单词，是小块文本（单词/片段/标点/控制符）。决定上下文窗口大小、KV 缓存大小、延迟和预填充时间。

🏗️

Transformer

骨架。Token → 向量 → Attention → MLP → 输出投影。堆叠数十层。RoPE 保持位置信息，残差连接稳定深层网络。

⚡

Attention

决定哪些早期 token 重要。MHA 灵活但缓存大，GQA 是中间立场，MQA 省内存。FlashAttention 等内核可显著加速。

💾

KV Cache

隐藏的内存账单。大小 ∝ token × layer × kv_heads × head_dim × precision × 2。13B MHA 在 32K 时可消耗 16 GiB。

🔄

预填充 vs 解码

预填充处理提示（可并行，计算受限），解码一次一 token（带宽受限）。长提示 → 慢预填充。长答案 → 慢解码。长对话 → 都慢。

🎛️

量化

降低精度以节省内存。FP16 → Q8(几乎无损) → Q6/Q5(优质) → Q4(消费级甜区) → Q2(必须)。数学/代码/工具调用最敏感。

📖 详细分析

🔤 Token — 模型的工作单位

LLM 不读原始文本，他们看到 token：内部表示为整数 ID 的小块文本。

常见类型：完整单词、单词片段、标点、空格前缀、字节级回退、特殊控制标记。

影响面：

  • 上下文窗口能装多少文本
  • KV 缓存变得多大
  • 预填充时付出多少延迟
  • 多语言/代码的压缩效率
词汇量大词汇量 → 更少 token → 更快
上下文窗口2026: 8K~1M token。但支持 ≠ 同等质量
测试建议始终测试实际使用的上下文长度
🏗️ Transformer — 骨架

大多数现代 LLM 是基于 Transformer 的 仅解码器架构：预测下一个 token，回看所有之前的 token。

简化层包含：

  1. Token 嵌入：Token ID → 向量
  2. 位置信息（RoPE）：通过旋转表示编码位置
  3. 自注意力：每个 token 回看前置 token，决定什么重要
  4. MLP/前馈块：展开并压缩表示的密集非线性计算
  5. 层归一化 + 残差连接：稳定深度网络
  6. 输出投影：最终隐藏状态 → 词汇表上的 logits
👁️ Attention — 谁重要？

Attention 是 token 决定哪些早期 token 对下一个预测重要的机制。也是本地推理对内存敏感的关键原因。

类型说明KV 缓存大小表达能力
MHA每头独立 K/V最大✅ 最强
GQA查询头分组共享 K/V中等✅ 好
MQA多查询头共享一个 K/V最小⚠️ 可能较弱

FlashAttention/SDPA 等内核减少注意力内存流量，使 GPU 更忙。同样模型+硬件下，好内核可以快得多。

比较模型时：看注意力类型 + KV 头 + 上下文长度 + 运行时支持，而不是只看参数数量。
💾 KV Cache — 隐藏的内存账单

KV 缓存是模型生成时的工作内存。存储之前 token 的 K/V 注意力状态，避免每生成一个新 token 就重新计算整个历史。

大小公式：token × layer × kv_heads × head_dim × precision × 2

模型4K 上下文32K 上下文说明
7B MHA (FP16)~2 GiB~16 GiB旧式设计
7B GQA (FP16)~0.5 GiB~4 GiB现代设计
7B GQA (FP8)~0.25 GiB~2 GiB推荐压缩层
这就是为什么模型空提示时能跑，加载长文档时崩了。权重合适，但工作内存不合适。
🔀 预填充 vs 解码 — 两个性能机制
预填充 (Prefill) 处理提示阶段。20K token 文档需先全部处理才能生成第一个 token。相对可并行化，GPU 能有效处理。
解码 (Decode) 一次生成一个新 token。每个 token 依赖序列，因此更连续性。通常是感觉快慢的决定性阶段。

三个惩罚规则：

  • 长提示 → 惩罚预填充
  • 长答案 → 惩罚解码
  • 长对话 → 两者都惩罚（KV 缓存不断增长）
🎯 解码 — Logit 到 Token 的最后一公里

模型生成 logits 后还没写任何东西。解码是「将分数 → 选中一个 token → 追加到序列 → 重复」的策略。

三个控制维度：

  • 随机性：允许多少变化？(temperature)
  • 尾部覆盖：采样器能深入低概率 token 多远？(top-p)
  • 边界：什么防止循环/漫无目的/失控？(repetition penalty, stop tokens)
精确工作低温 + 短 max_tokens + 约束解码
创意工作高温度 + top-p + 多候选排序
编码工作第一遍保守，探索阶段放宽
贪婪解码并不总更准确。它通常很脆弱，可能陷入循环或产生通用答案。

🛠️ 基础设施决策

📦 模型包包含什么

一个可运行的本地 LLM 远不止一个大权重文件：

  1. 架构/配置：层数、隐藏大小、注意力类型、RoPE 设置
  2. 权重：学习参数，存储为 safetensors / GGUF / GPTQ / AWQ / EXL2
  3. Tokenizer：文本 ↔ Token ID 的规则
  4. 聊天模板：system/user/assistant/tool 消息的确切 token 格式
  5. 生成配置：temperature, top-p, stop tokens 默认值
  6. 许可证 & 模型卡：法律和操作说明
如果 tokenizer/聊天模板错误，同样的权重会感觉像坏了。
📐 量化 — 精度的艺术
层级说明适用场景
FP16/BF16质量最佳基线VRAM 充足时的评估基准
Q8/INT8几乎无损，仍然较大VRAM 够用，质量优先
Q6/Q5优质，节省适中强力的中间立场
Q4消费级默认甜区大多数聊天和文档工作流
Q3/Q2仅当必须塞更大模型数学/代码/结构化输出先下降
高精度的小模型可以击败压太死的大模型。Q6 的 7B 能在推理任务上击败 Q2 的 13B。
💻 VRAM 数学 — 三巨头

三个主要消费者：

  1. 模型权重：参数 × bytes_per_param（FP16 ≈ 2B/param, Q4 ≈ 0.5B/param）
  2. KV 缓存：随上下文线性增长
  3. 运行时开销：CUDA 开销、框架缓冲区、临时张量

现实估算：total = quantized_weights + KV_cache + runtime + batch + safety_margin

陷阱：Q4 的 13B 在 8K 上下文下轻松，但在 32K 下崩了。权重没变，上下文变了。留出 10-20% 余量。

🚀 什么控制速度？

TPS 不受一件事控制：

内存带宽解码通常受带宽限制（反复传输权重）
GPU FLOP预填充和大批量更依赖计算
CPU 溢出痛苦的设置：性能可能崩溃
推测解码DFlash/DDTree 加速生成

🎯 模型选择

📋 模型选择五项检查
  1. 任务适合：聊天/编码/文档/代理/多模态/边缘/微调
  2. 内存适合：权重 + KV 缓存 + 运行时开销 + 余量 ≤ 可用内存的 80-90%
  3. 接口适合：tokenizer, 聊天模板, stop tokens, 工具模式
  4. 运行时适合：运行时是否良好支持这种架构/量化/上下文
  5. 许可证适合：你实际能在计划使用的地方用吗？
实际问题不是"最好的模型是什么"，而是"最小的模型是什么在硬件上赢了实际工作负载"。

🔒 安全与隐私

🛡️ 本地 ≠ 自动安全

本地并不自动意味着安全。威胁包括：恶意模型文件、pickle 权重加载、prompt 注入、日志泄露、遥测外传。

可行基线：

  • 优先 safetensors 或 GGUF，避免不受信任的 .bin
  • 非特权用户 + 容器沙箱 + 可禁用网络
  • 凭证排除在提示和 RAG 索引外
  • 版本追踪：模型、提示、适配器、运行时、量化
🤖 本地 Agent 需要护栏
1
界定范围 — 只给代理实际需要的目录、API、网络、凭证
2
约束执行 — 沙箱、容器、最小权限用户、破坏性操作需确认
3
输入即恶意 — 检索的文档、网页、工单可能包含 prompt 注入
4
审计追踪 — 记录工具调用、模型版本、提示、批准

📚 RAG 与文档工作

🔍 RAG 击败巨型提示

RAG = 检索增强生成。从知识库检索相关块，只把那些块给模型。

好 RAG 系统：文档提取 → 解析 → 分块 → 嵌入 → 向量索引 → 检索 → 重排序 → 提示构建 → 生成 → 基础检查 → 评估

坏 RAG 不是 LLM 的锅 — 坏在分块、检索、重排序和评估。

✅ 本地 LLM 运行手册

📋 信任一个本地模型前的最后检查
1
选择并适合 — 选模型家族，读许可证，确认硬件，选量化，估算全部内存费用
2
加载和格式 — 优先 safetensors/GGUF，验证 tokenizer 和聊天模板
3
评估和操作 — 用代表性提示测试，测 TTFT 和解码速度，跟踪峰值内存
4
版本控制一切 — 模型、量化、运行时、提示、模板、评估集、硬件配置

⚖️ 开放权重 ≠ 开源

2026 年"开放模型"一词常被草率使用：

  • 开放权重：可以下载权重。不自动意味着可商用/可修改
  • 源可用：代码或权重可见。不一定是开源许可证
  • 开源 AI (OSI 定义)：架构+参数+推理代码+数据信息
规则：商用任何模型前，阅读模型卡和许可证。

本地 LLM 主要是内存数学 + 格式化 + 评估

你不需要神话来很好地运行本地模型。你需要知道内存中适合什么、模型期望哪个模板、运行时的行为方式。
